adapters/src/file_persistence.cpp:122-136: CRITICAL - save() picks the overwrite target by the on-disk SEQ WORD ALONE, not by whether that file's blob actually loads. read_record() marks a file with an intact 12-byte header but a TORN/truncated body as present=true with its (higher) seq (see lines 62-76). So after the very crash-mid-write this design exists to survive — file A: seq=5 VALID, file B: seq=6 header-intact-but-torn — the next save() computes write_to_a=(sa<=sb)=(5<=6)=true and OVERWRITES the last-good A (next_seq=7). A crash during THAT write leaves A torn and B torn => BOTH files unusable, last good state LOST. The crash-safety invariant ("load() always recovers the last good state", file_persistence.hpp:6-7) is violated. The test at test_file_persistence.cpp:170-173 even exercises exactly this overwrite-the-good-file step (A=1 valid, B=2 torn -> save writes seq 3 to A) and only passes because no crash is simulated during it. Fix: pick the inactive target by which file load() would NOT choose — i.e. run the core deserialize (or at least a checksum check) on each candidate during save() and overwrite the file that FAILS to load (or, if both load, the genuinely lower-seq one); never overwrite the highest-seq file whose blob the core accepts. Equivalently: the "active" file to preserve is the one load() returns, not the one with the larger seq header.

adapters/src/file_persistence.cpp:18-22 (hpp doc): MAJOR - the header contract "a crash mid-write leaves at most the INACTIVE file torn and load() always recovers the last good state" / "writes to the LOWER-seq file" is only correct when the higher-seq file is GOOD. A higher-seq TORN file (the realistic crash residue) makes the lower-seq file the last-good one, and save() then targets it. Doc overstates the guarantee. Fix: reconcile the doc with the CRITICAL fix (target = the file load() rejects), or explicitly document that a torn higher-seq file is a known unrecovered case.

adapters/src/config_loader.cpp:236-238: MINOR - per-sensor `id` and the `[sensor.N]` section index are independent and neither duplicate-id nor id-vs-index agreement is checked; two sections can set the same `sc.id`, and the core validate() (estimator.cpp:1643 "TODO: per-sensor checks") does not catch it. A duplicate/typo'd sensor id parses + validates Ok and silently mis-binds at add_source/restore time. Fix: after parsing, reject duplicate SensorConfig::id (and optionally a `reference_sensor_id` with no matching sensor) in parse() before returning Ok.

adapters/src/config_loader.cpp:257-259: MINOR - `is_reference` (per-sensor) and `reference_sensor_id` (global) can disagree (e.g. is_reference=true on sensor 1 while reference_sensor_id=0) with no diagnostic; the loader sets both independently and core validate() only range-checks reference_sensor_id. Fix: cross-check that the sensor flagged is_reference matches reference_sensor_id (or document that reference_sensor_id is authoritative and is_reference is advisory).

adapters/src/threaded_estimator.cpp:64-110: MINOR - drain_and_stop() after a prior stop() takes the inline-drain `else` branch (worker already joined -> not joinable) and drains the queue inline even though stop_now_ is already true; the inline loop never checks stop_now_, so "stop then drain_and_stop" processes leftover stamps that a plain stopped wrapper would have dropped — a small contract inconsistency (stop() drops pending; the post-stop drain_and_stop resurrects them). Not a race. Fix: in the inline-drain branch, skip draining when stop_now_ was already set, or document that drain_and_stop() always honors the queue regardless of a prior stop().

adapters/src/config_loader.cpp:57-69: NIT - parse_int uses std::stol; a value exceeding long range throws std::out_of_range, caught and reported as the generic "expected int" rather than an out-of-range message. Cosmetic (still a hard InvalidConfig). Fix: optionally distinguish overflow in the error text.

adapters/src/file_persistence.cpp:84-85 (comment): NIT - the comment concedes flush()+close() is only "the std-portable analogue" of fsync and a real ECU adapter would fsync the fd. DESIGN §7 specifies "write inactive -> fsync -> flip"; std::ofstream flush does not guarantee the bytes reached stable storage (OS page cache may still hold them), so a power-loss window remains even with the CRITICAL fixed. Acceptable for the relaxed-edge subset but should be called out as a deferred durability gap (13b / platform layer). Fix: note the fsync gap in ISSUES/HANDOFF as deferred.
