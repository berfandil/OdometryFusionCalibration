## Deviation judgments

- DEVIATION 1 (veto normalization: leave-one-out instead of the spec'd full-set RMS) — APPROVED, algebra VERIFIED: with the full-set spread s² = Σw_j d_j²/Σw_j, d_i²/s² = d_i²·Σw/Σw_j d_j² ≤ d_i²·Σw/(w_i d_i²) = Σw/w_i, so d_i/s ≤ sqrt(Σw/w_i) = sqrt(n) at equal weights — strictly below 3.0 for n ≤ 9 (equality only in the degenerate all-other-d=0 limit), and a HIGH-weight outlier (the dangerous case) is bounded even tighter. The spec'd normalization would have made the veto dead code at every realistic source count; LOO is the minimal normalization that keeps the fixed 3.0 constant live. The implementer's claim is correct as stated. HOWEVER the LOO form introduces a degenerate-zero hair-trigger the full-set form could not have (MAJOR finding 1 below) — approval is conditional on fixing it.
- DEVIATION 2 (cov-cal sweep extended past the spec'd brace to q_scale=3.0 instead of STOP) — APPROVED with a reservation: methodology VERIFIED identical to the coupled calibration (same 3-source mounted rig, same noise params 0.02/0.005, same nees_traj/turning/straight set, M=30, warmup 20, same NEES estimator); the recorded table is strictly MONOTONE decreasing (worst 30.10 → 21.65 → 15.24 → 10.20 → 7.67 → 6.14 → 5.12 → 3.85), so 3.0 is the unique never-overconfident + above-floor point (4.0 trips the floor at turning 1.06); the in-band claim holds on EVERY trajectory (5.12 / 1.41 / 3.30, all < 5.6 and > 1.2, worst 5.12 ∈ [4.0, 5.6]) and the permanent guard asserting exactly this ran green in this review's gate re-run; the band was NOT weakened (coupled guard file untouched, split cap 5.6 = the spec's band cap) and the 0.7-is-wrong-for-split fact is pinned loud (nees > 12.0 at 0.7, measured ~21.7); the root cause (the coupled metric put the MIXED spread on all six axes, padding each channel's Q with the other channel's scatter — removed by honest per-channel sizing) is coherent and the veto was exonerated by the veto-off probe (21.1 ≈ 21.65). Judgment: extending was the right engineering call — the STOP rule's protected invariants (band integrity, no out-of-band ship) are both honored, and stopping would have shipped nothing while the root cause was already in hand. The reservation: the spec literally said STOP and report, so the user should ratify this post-hoc; and the extension created an undocumented config trap (MAJOR finding 2). The "blocks balanced 2.21/2.70 → no per-channel scale knob" inference is sound per §1.3's own criterion (the band is met with one shared knob; a second knob could only re-balance ~20% between blocks that are both already in the safe regime).

## Findings

- src/core/median.cpp:418: MAJOR: the LOO veto threshold has no floor — when the OTHER inputs in a channel are (near-)coincident, loo → 0 and the flag degenerates to `d_i > ~0`, so ANY honest deviation (sensor noise, quantization) flags the source every step and scales its OTHER-channel weight ×0.1 with a re-solve. This is not a corner case for this project: the slice's own target rig (KAIST urban, recommended config split+veto-ON) runs sources that SHARE wheel translation — two near-identical aligned translations make the third source's translation LOO spread ~0, and that third source is the FOG-grade heading source whose ROTATION weight then gets permanently vetoed to 0.1× by trivial translation deviations, defeating the slice's purpose under its own default policy. The "does NOT fire on clean quality differences" test uses comfortable scatter on all sources and cannot see this. Fix: floor the LOO spread (e.g. flag only when loo > 0 AND d_i exceeds max(kVetoNormDist·loo, an absolute per-channel floor tied to p.eps or a new fixed constant), or blend with the full-set spread) and add a coincident-inlier regression test (two identical translations + one noisy source: veto must NOT fire); re-check the real-data plan with veto ON afterward.
- include/ofc/core/config.hpp:165: MAJOR: enabling split_median with the DEFAULT q_scale=0.7 yields a grossly overconfident filter (NEES ~21.7 vs DOF 6 — pinned by the guard test), and the q_scale≈3.0 requirement exists only in test-file comments and the commit message: validate() accepts the combination silently, the loader's kKnobDoc line for split_median (adapters/src/config_loader.cpp:134) says nothing, and CONFIG docs are deferred to a follow-up. Anyone flipping the bool on a real rig — including the slice's own post-merge real-data validation if the orchestrator copies the "recommended urban config + split_median" line from the spec, which also does not mention q_scale — ships a miscalibrated filter. Fix: at minimum extend the kKnobDoc split_median line and the config.hpp field comment with "requires q_scale ≈ 3.0 (see test_cov_calibration_split.cpp)"; consider making the orchestrator's CONFIG doc update a merge condition rather than a follow-up, or a validate()-level rejection/warning of split_median + q_scale < 1.5.
- src/core/estimator.cpp:2228: MINOR: validate() allows rot_weight_prior = 0 with the documented semantics "never trust this source's rotation", but if ALL sources carry 0 the rotation channel's weight set is all-zero and ChannelWeights::bind falls back to UNIFORM — every source's rotation is trusted equally, the exact opposite of what each per-sensor 0 declared. The solver-level fallback is tested; the contradictory all-zero CONFIG semantics is neither rejected, documented, nor tested (no estimator-level rot_weight_prior=0 test at all). Fix: reject the all-sources-zero rotation prior in validate() (or document the uniform fallback at SensorConfig::rot_weight_prior) and add the single-source prior=0 estimator test.
- src/core/estimator.cpp:63: MINOR: the static_assert ties kMaxSourcesCap only to Eskf::kMaxMedianInputs; median::kMaxSplitInputs (median.hpp:99) is a third copy of 32 with no compile-time tie, and solve_split's behavior past the cap is the nastiest kind of silent: the base solves still use all n inputs but the veto silently disables (n <= kMaxSplitInputs gate) and w_rot_final/w_trans_final are filled only to nc < n — a future cap bump on the estimator side would hand median_influence_split garbage tail weights. Fix: add `static_assert(kMaxSourcesCap <= median::kMaxSplitInputs, ...)` next to the existing one.
- tests/test_split_median.cpp:379: MINOR: acceptance §2 item 6 says EVERY calibration DOF converges in its regime under the split consensus — the split-ON observability tests cover yaw/pitch+scale (straight) and lever (turn) with honest negatives, and exclude the time-offset DOF with a stated valid rationale (time-sync samples raw source ‖ω‖, never the median), but the ROLL DOF is silently absent (the roll histogram is even configured in the test rig). Fix: add the roll-regime positive/negative pair or state the exclusion rationale like the offset one.
- tests/test_split_median.cpp:446: MINOR: the split+multi_bias estimator case is a clean-source smoke (veto never fires), so the split + multi_bias + veto-ACTIVE interaction is covered only by the unit-level FD pin in test_multi_bias.cpp — no end-to-end sim ever runs a vetoed source through the block-diagonal coupling into F over many steps (where a wrong veto-scaled Ω would show up as bias misattribution or covariance corruption rather than a one-window FD mismatch). Fix: one estimator-level case with a both-channel-faulty source + bias_states + split: assert finite state, bias stays ~0 unobservable, and the fault stays rejected across the run.
- src/core/median.cpp:436: MINOR: the all-sources-flagged edge is unhandled-by-design but untested: if every source trips the veto in one channel, ALL the other channel's weights scale ×0.1 and the re-solve is identical by Weiszfeld scale-invariance (wasted WCET, harmless result) — nothing pins that harmlessness, and with the finding-1 hair-trigger this edge is reachable. Fix: cheap test (3 mutually-distant sources where every LOO flag fires) asserting the re-solve equals the base solve and weights report the scaling.
