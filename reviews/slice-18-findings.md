## Findings

- tests/test_multi_bias_sim.cpp:298: MAJOR: the relaxed acceptance-2 thresholds rest on a "linearization-radius plateau" diagnosis that has NO independent asymptotic cross-check anywhere in the suite: no test shrinks the planted bias (or the source noise) and pins recovery fraction → 1. The discriminator matters because a systematic ~15–20% scale error in the coupling chain (a wrong dt convention at one of the three dt sites, a transport factor, an H mis-model) produces exactly the same "plateau" signature and passes every committed threshold — the production==pin test only proves production equals the test-file reference, and the FD pin only verifies the per-window block, not the accumulated filter-level scale. The supporting evidence ("doubling the learn phase left the values at this plateau", the 1.0/0.01 prior sweep) lives only in comments, reproduced nowhere. Fix: add one asymptotic case — same rig, planted bias ÷10 (or noise ÷10, pushing bias·dt well inside the claimed radius) — and pin recovery > 95%; if it plateaus at ~85% there, the diagnosis is wrong and a scale bug is live.
- tests/test_multi_bias_sim.cpp:304: MAJOR: the separation assertions bound only 2 of 6 bias DOF, and the measured run (MESSAGE output, re-run by this review) shows the unchecked DOF carrying junk AT OR ABOVE the planted signal: |b0| = 0.171 > |planted| = 0.158; the planted source's own spurious ω_x = 0.080 rad/s exceeds the planted ω_z = 0.05; b0's v_z = −0.124 is 83% of the planted v_x. "Co-sources ≤ 0.5× planted" is true only per-driven-DOF — by norm the co-source biases are NOT "≈ 0" in any sense, and during a coast every one of those spurious components actively de-biases its source (a 0.08 rad/s phantom roll-rate correction). The sim coast still wins 4.4×/14× because the junk partially cancels through the median, but nothing pins that cancellation. Fix: bound the undriven-DOF junk explicitly (e.g. |b_i,undriven|∞ < some fraction of the prior σ) or at minimum bound |b0|/|b2| full norms, and state the per-DOF-only separation honestly in the test header and the orchestrator report.
- src/core/estimator.cpp:499: MAJOR: kMultiBiasCov0 = 0.04 is load-bearing for STABILITY, not just the acceptance-2 ratio — by the comment's own mechanism the safe prior scale is set by the linearization radius bias·σ·dt vs the per-window noise scale d_i, which is a RIG-DEPENDENT quantity (sensor noise floors, window length, source count), yet it is a compile-time constant hand-tuned to this sim rig (noise 0.01/0.002, 50 Hz, 3 sources). On urban12 — the slice's actual target, different noise floor and cadence — 0.04 may sit outside the radius and trigger the self-sustaining-junk mode the comment itself warns 1.0 exhibits, with no knob to retune without a rebuild. Neither claimed failure extreme is pinned by a test (no case shows 1.0 blowing up co-sources, none shows 0.01 yielding recovered/planted == bias_observable). Fix: promote to a Config knob (default 0.04) hashed like bias_process_noise, or derive it from q_floor/window noise; add a cheap pin of at least the 1.0-divergence claim so the constant's justification is regression-guarded.
- tests/test_multi_bias_sim.cpp:136: MAJOR: every flag-ON sim acceptance disables Slice-9 reliability (floor == cap == 1.0) to dodge a diagnosed adverse feedback loop (wrongly-de-biased co-source scatters → down-weighted → loses median influence → its bias error becomes LESS observable → persists) — but the recommended real-data config runs reliability ON, so the deployment configuration has ZERO acceptance coverage and the known interaction is documented only in this test comment. Isolating the mechanism is legitimate test design; shipping with no coverage of the combination is not. Fix: add one reliability-ON separation/coast case with explicitly weaker (documented) bounds — even "no divergence + coast still better than baseline" — and record the interaction in config.hpp at multi_bias_enabled (or DESIGN), not in a test comment the orchestrator's real-data run won't read.
- tests/test_multi_bias_sim.cpp:574: MINOR: the flag-ON NEES test asserts only never-overconfident (< 5.5) + a token floor (> 0.05); the measured 2.58 is BELOW the cov-cal consistency band [4.0, 5.6] — the flag-ON pose marginals are ~1.9× pessimistic, a real consistency regression in the direction the Slice-14 saga spent effort eliminating. ## Implementer deviations — explicit judgments

- (1) Relaxed acceptance-2 thresholds (~80–90% recovery, co-source ≤ 0.5× planted) + kMultiBiasCov0 = 0.04: PARTIALLY ACCEPTABLE. The relaxation itself is honestly documented at the assertion site and the recovered numbers are real (re-run: b1 v_x 0.118/0.15, ω_z 0.056/0.05, coast 4.3×/14×). NOT acceptable as-is: the plateau diagnosis is unfalsified (MAJOR 1), the co-source claim is true only on the driven DOF (MAJOR 2), and the prior constant is a rig-dependent stability parameter shipped as a compile-time magic number with its justifying sweep unreproduced (MAJOR 3). The deviation is plausibly genuine physics — weak observability of bias DIFFERENCES through an influence-weighted sum is a real structural limit — but the evidence committed does not yet distinguish it from a scale bug.

The FD pin (def0cee) was a genuine red-gate executed BEFORE any production code — it stopped the slice and forced the design amendment, which is the strongest possible red. The claimed mutations (zeroed coupling → 7 separation assertions fail; skipped de-bias → coast ON == OFF) are credible because the production==pin test at 1e-9 plus the end-to-end thresholds would catch both; the process deviation is documented, not hidden.
- (3) Reliability EMA disabled in sims: JUDGED MAJOR (finding 4) — isolation is good test design, but the deployment combination ships with zero coverage and the diagnosed feedback loop is recorded only in a test comment.
- (4) Loader per-sensor bias_states/bias_process_noise keys: ACCEPTABLE and overdue (the keys existed in core since 11b but were unparseable from config files — a real gap this slice needed for real-data runs). Parse/reject/default coverage is thorough (adapters/tests/test_config_loader.cpp:367-435, incl. negative-pn → core OutOfRange and the flag-off guard surfacing through validate()); knob-doc string updated. Verified green.
